# Main/Dev 分支流程

本文档只规定 `main` 和 `dev` 的长期职责。代理行为约束以仓库根目录
`AGENTS.md` 为准，算法和接口说明由 `CODEX_CEPH.md` 维护。

## `main`

`main` 是可构建、可部署和可打版本标签的生产分支，只保留已经确定的方案：

- Heat Predictor 生产代码及必要的 Ceph OSD/MGR 适配。
- 当前有效的实现说明和操作手册。
- 生产构建需要的配置。

`main` 不保留 Trace、测试探针、实验 profile、多套算法分支、TODO、实验计划或
原始报告。

## `dev`

`dev` 是日常开发和验证分支，以当前 `main` 为生产基线，可以额外保留：

- 尚未发布的生产改动。
- Trace 采集、转换、replay 和离线分析能力。
- 算法、性能、并发和正确性测试。
- 活动 TODO、设计、实验计划和开发记录。

## 开发与发布

1. 日常开发、调试和负载验证默认在 `dev` 进行。
2. 生产改动与 dev-only 的 Trace、测试和文档应拆成独立提交。
3. 新方案在 `dev` 验证通过后，只选择生产代码和稳定文档发布到 `main`，不得整体
   merge `dev`。
4. `main` 完成生产构建和必要的集成检查后才能打版本标签，标签只指向 `main`。
5. 发布后以新的 `main` 为生产基线重新对齐 `dev`，继续保留有效的开发能力。
6. 直接在 `main` 修复的问题也必须同步回 `dev`。
7. 未经用户在当前任务中明确要求，不执行 commit、rebase、merge、tag 或 push。
